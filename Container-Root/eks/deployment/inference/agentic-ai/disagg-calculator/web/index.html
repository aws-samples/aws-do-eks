Aggregate vs Hybrid vs Disaggregate — what the verdict means

Every LLM request has two phases: prefill (read the whole prompt, compute-bound, bursty) and decode (emit one token at a time, memory-bandwidth-bound, steady). The only question this tool answers is where you run those two phases relative to each other. There are three choices:

■ Aggregate

Prefill and decode share the same GPUs. A request prefills, then decodes, in place. No KV cache transfer. A prefill burst can stall the decode stream of co-batched requests.

Picked when: prefill is cheap vs decode (little interference to remove), or the KV-transfer toll would cost more than it saves. Cheapest $/token at low–moderate load.

◆ Hybrid

The middle verdict — hybrid when 0.30·(1−cpr) ≤ fav < 0.55·(1−cpr), i.e. 0.090 ≤ fav < 0.165 at the default chunked-prefill recovery cpr=0.7. There is prefill→decode interference worth mitigating, but a full split isn't justified: either the KV-transfer toll is too high or the imbalance is only moderate.

In practice it's TaiChi's differentiated P/D regime — prefill is chunked and interleaved with decode on the same GPUs so a prefill burst can't monopolize a step (interference bounded), with no cross-node KV hop. This is exactly what chunked_prefill_recovery=0.7 models.

Picked when: in between — enough prefill interference to want separation, but not enough (or too high a KV toll) to justify a full disaggregated split.

▲ Disaggregate

Prefill and decode run on separate GPU pools; the KV cache is shipped prefill→decode over the fabric (EFA/NIXL RDMA). Decode never sees a prefill burst → clean, steady latency.

Picked when: prefill bursts would stall many decode steps under co-location (high interference) and KV transfer is cheap. Wins on SLO-attainment / tail latency, and on $/token only at scale.

How the verdict is computed (the actual formula)

It is one number, fav = toll × interference, both derived from first-order rooflines in analyze/calc.py (mirrored in this page's JS — they're kept identical):

interference = 1 − exp(−imbalance / 4) # imbalance = prefill_ms / decode_ITL_ms # → how many decode steps one prefill burst stalls toll = 1 / (1 + 4 × transfer_overhead) # transfer_overhead = KV_transfer_ms / decode_phase_ms # → 1 when shipping the KV is ~free, →0 when it's expensive fav = toll × interference fav ≥ 0.55 → DISAGGREGATE (lots of interference to remove, cheap to ship KV) fav ≥ 0.30 → HYBRID (in between) else → AGGREGATE (prefill cheap, or KV too expensive to move)

Intuition: disaggregation only pays off when it removes a lot of decode-stalling interference AND the KV cache is cheap to move. Long prompts + short outputs raise interference (favors disagg); short prompts, or a fat KV cache over a slow link, raise the toll (favors agg). The verdict is a sliding scale, not a coin flip.

Under the hood the two inputs are not naive rooflines — prefill_ms adds the O(ISL²) quadratic-attention term (dominates at long context, not just linear 2·P·ISL) scaled by the quantized peak (fp8 ≈2×, fp4 ≈4× vs bf16); decode_ITL adds the context-dependent KV-read (each token re-reads the whole KV cache so far) on top of the measured family calibration; and chunked-prefill recovery, the hybrid-Mamba constant SSM state, and an SLO gate are all folded in. Every one mirrors analyze/calc.py exactly (parity-tested).

What happens as you scale the number of nodes?

“Adding nodes” means two different things, and they behave oppositely. The repo models both (analyze/node_scaling.py and analyze/pd_pool_scaling.py):

1. Replica scaling — model fits, add independent copies

Throughput grows ~linearly with nodes; per-request latency stays flat; $/token stays flat.

Measured-shape example (550B/p5en): 1→8 nodes ≈ 3,222 → 23,712 tok/s (7.4×, router-eff ~0.92). This is the honest "just add servers" law.

2. Prefill/Decode pool scaling — the at-scale agg-vs-disagg crossover

Disaggregation's fixed costs amortize as you add nodes: the integer prefill:decode split gets finer, so wasted capacity shrinks.

550B fragmentation penalty: 49% @ 2 nodes → ~1% @ 32 nodes. So there's a computable crossover: small models flip to disagg at 1–2 nodes; frontier (8-GPU/replica) models around 5–7 nodes.

The at-scale law, in one line: in the work-conserving limit aggregation and disaggregation push the same raw throughput — so disaggregation's only structural win (removing decode interference) is fixed, while its costs (KV toll + P:D fragmentation) shrink with scale. ⟹ the bigger the fleet, the more disaggregation wins.

Honesty: the crossover floor (agg wins at the 1P:1D minimum) is MEASURED on two model families; the rising slope beyond it is a calibrated PREDICTION (live confirmation is blocked by EFA capacity on the shared cluster). Node-scaling is computed in the Python tools; this interactive calculator currently shows the per-replica verdict + cost knee below.

What this tool is — and what it is not

✓ It IS

  • A fast, transparent first-pass screen (agg / hybrid / disaggregate) from model architecture + workload.
  • Calibrated to real measured anchors (550B/p5en ITL, KV-over-EFA transfer penalty) — see the badges below.
  • An auditable derivation: every term is a first-order roofline you can read and challenge.

✗ It is NOT

  • A benchmark or a validated simulator — predicted numbers are calibrated, not measured.
  • A multi-tenant / load-dependent goodput model (batching, queueing, EP all-to-all are out of L0 scope).
  • Generalizable off the anchor without re-calibration — the n=1 family correction is honest, not universal.
