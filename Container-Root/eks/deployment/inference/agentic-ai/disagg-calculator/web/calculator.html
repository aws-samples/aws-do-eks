Does disaggregation help here?
—
Each axis is a speedup = aggregation ÷ disaggregation: >1 = disagg is better, <1 = disagg regresses, within ±10% = no-op. Measured proof both outcomes are real: 550B short-chat regresses on cost ~0.5× (agg won the live A/B 2.1×); a long-context agent workload helps on ITL. At the 1P:1D floor disagg is a latency play, not a cost play — cost tops out ~0.59× (2× GPUs is a hard 2×), TTFT always pays the KV hop, ITL is the win. The cost win needs P:D right-sizing (pd_pool_scaling.py), not this floor — so an overall “HELPS” is an at-scale / right-sized outcome, not a 1P:1D-floor one (at the floor you trade cost+TTFT for ITL → MIXED).
Honest by design — across our 240-cell catalog the verdict splits roughly evenly: ~52% disaggregate, ~48% aggregate-or-hybrid (reproduce: tests/catalog_distribution.py). On cost at the 1P:1D floor, disagg almost always regresses — a no-op or a regression is a real result, not a failure.

What this tool is — and what it is not

✓ It IS

  • A fast, auditable first-pass screen that derives agg / hybrid / disaggregate from the model architecture.
  • Calibrated to real measured anchors (550B/p5en ITL 82ms, KV-over-EFA ~127ms hop).
  • Every term a first-order roofline you can read in the derivation chain below and challenge.

✗ It is NOT

  • A benchmark / validated simulator — every "predicted" number is calibrated, not measured.
  • A batched, load-dependent goodput model (queueing, EP all-to-all are out of L0 scope).
  • Generalizable off the anchor without re-calibration — the n=1 family correction is honest, not universal.

What the three verdicts look like

Every request has two phases — prefill (read the whole prompt, compute-bound, bursty) and decode (emit one token at a time, memory-bandwidth-bound, steady). The verdict is where you run those two phases relative to each other.

■ Aggregate

Prefill + decode fully co-located on the same GPUs. No KV transfer. A prefill burst can stall the decode stream of co-batched requests.

Picked when: prefill is cheap vs decode (little interference), or the KV toll would cost more than it saves. Cheapest $/token at low–moderate load.

◆ Hybrid

The middle verdict — hybrid when 0.30·(1−cpr) ≤ fav < 0.55·(1−cpr), i.e. 0.090 ≤ fav < 0.165 at the default cpr=0.7. Interference is real but a full split isn't justified (KV toll too high, or imbalance only moderate).

TaiChi's differentiated P/D regime: prefill is chunked and interleaved with decode on the same GPUs so a burst can't monopolize a step — no cross-node KV hop. Exactly what chunked_prefill_recovery=0.7 models.

Picked when: in between — enough interference to want separation, but not enough (or too high a KV toll) for a full split.

▲ Disaggregate

Separate prefill + decode pools; the KV cache is shipped over EFA / NIXL RDMA. Decode never sees a prefill burst → clean, steady latency.

Picked when: prefill bursts would stall many decode steps (high interference) and KV transfer is cheap. Wins on SLO-attainment / tail latency, and on $/token at scale.

⚠ One word, two meanings — don't confuse them. “Hybrid” above is a serving topology (the middle verdict). Separately, “hybrid-Mamba” is a model architecture (e.g. Nemotron-3 Ultra 550B — attention + Mamba layers), selectable in the family dropdown. They are unrelated: a hybrid-Mamba model can be served with an aggregate, hybrid, or disaggregate topology.

Favorability vs context (ISL sweep)

Disagg-favored where the curve crosses the (chunked-prefill-rescaled) disagg threshold. Non-monotonic: drops again at extreme prefill+short output (KV toll un-amortized).

Cost vs concurrency — find the knee

$/Mtok falls until the goodput knee (KV-budget / batch ceiling), then flat. Past it, TTFT explodes for no throughput gain.
